The Architecture: Stack, Memory, and Storage

The EVM has three places to put data. If you can't explain the difference, you will write expensive, broken contracts.

1. The Stack (1024 slots max)

LIFO data structure. Every arithmetic op, comparison, and function argument passes through it. Each slot: exactly 256 bits (32 bytes).

PUSH1 0x03   → Stack: [0x03]
PUSH1 0x05   → Stack: [0x05, 0x03]
ADD           → Stack: [0x08]
// Everything on the stack is padded to 32 bytes. No "int8" exists on stack.

Gas: 3 per operation. Effectively free. Moral: Do as much computation on the stack as possible. JS analogy: function locals — fast, temporary, scoped to execution frame.

2. Memory (Linear, Volatile, Per-Transaction)

Byte-addressable, linearly expanding. Starts empty each transaction, wiped after. Used for ABI encoding, Merkle tree hashing, computation buffers.

Gas: 3/byte write, 3/word read. Expansion cost is quadratic: words² / 512. JS analogy: allocating a large Buffer — persists for function call, then GC'd.

3. Storage (Permanent, Persistent, Expensive)

Persistent key-value store: 256-bit key → 256-bit value. Survives forever until overwritten. Every write = permanent blockchain state change.

SSTORE: 20,000 gas (new slot), 2,900 (warm overwrite). SLOAD: 2,100 (cold), 100 (warm). One new slot ≈ cost of an entire ETH transfer. JS analogy: writing to PostgreSQL — $0.001 per row. Cache reads in memory variables.

Storage Packing: Where 20,000 Gas Becomes 0

// ❌ WASTEFUL: 4 slots = 80,000 gas
uint8   a;  // Slot 0
uint8   b;  // Slot 1 ← NEW SLOT
uint8   c;  // Slot 2 ← NEW SLOT
uint256 d;  // Slot 3

// ✅ OPTIMIZED: 2 slots = 40,000 gas (50% saved)
uint8   a;  // ┐
uint8   b;  // ├─ Slot 0 (3 bytes, packed)
uint8   c;  // ┘
uint256 d;  // Slot 1 (uint256 ALWAYS starts new slot)

Packing rules: Variables pack in declaration order. Only types <32 bytes pack. uint256 always starts a new slot. Mappings and dynamic arrays always occupy a full slot pointing to their data — they break packing on both sides.

Storage Layout Breaking Changes

Adding a state variable to a deployed contract shifts every subsequent variable to the wrong slot. This has caused real exploits. Never change state variable order in a deployed contract. Zero exceptions.

Mental Model: The Restaurant Kitchen

Write order (sign) → waiter checks it (validate) → kitchen queue (mempool) → chef prioritizes (builder) → cooking (EVM execution) → bill (gas accounting). If food burns (revert), you pay for ingredients but get no dish.

4 Failure Modes JS Devs Miss

(1) Rejected by node — no gas. (2) Stuck in mempool — no gas but blocked nonces. (3) Included but reverted — you pay ALL gas used before revert. (4) Success. Most frontend bugs come from not handling 1–3.

Advanced Data Structures: Mappings, Structs, and Enums

Learning Objective

Understand how mappings, structs, and enums map to EVM storage slots, when to use each, and the gas implications of nesting them incorrectly.

Mappings: Not What You Think

In JavaScript, Map or {} stores key-value pairs in memory. Solidity's mapping is fundamentally different — it doesn't store keys at all. A mapping only stores a computation rule: given key K, the value lives at slot keccak256(abi.encode(key, slot)). This means:

  • You cannot iterate over a mapping (no .keys())
  • You cannot check if a key exists (unmapped keys return default value — 0 for uint, false for bool)
  • A mapping always occupies exactly 1 storage slot, regardless of how many entries it has
  • The "slot" a mapping occupies stores nothing — it's just the seed for the keccak hash
DataStructures.sol
pragma solidity 0.8.24;

enum Status { Pending, Active, Closed } // 0, 1, 2 — packed as uint8

struct User {
    uint128 balance;  // Slot 0 (16 bytes)
    uint128 locked;    // Slot 0 (16 bytes) — PACKED with balance
    Status  status;     // Slot 1 (1 byte)
    address referrer;   // Slot 2 (20 bytes)
}

contract DataStructures {
    mapping(address => User) public users; // Slot 0 — just the seed
    mapping(address => mapping(address => uint256)) public allowances; // Slot 1

    function register(address referrer) external {
        // Each User struct occupies 3 storage slots (0, 1, 2 relative to base)
        // base = keccak256(abi.encode(msg.sender, 0))
        users[msg.sender] = User({
            balance: 0,
            locked: 0,
            status: Status.Pending,
            referrer: referrer
        });
    }
}
Struct Packing Inside Mappings

When a struct is stored inside a mapping, packing rules still apply to the struct's internal layout. The balance and locked fields in the User struct above share a slot because they're both uint128 and declared adjacent. If you put a uint256 between them, packing breaks and you pay an extra 20,000 gas per registration. Rule: Order struct fields from smallest to largest to maximize packing.

Common Mistake: Default Values as "Exists" Checks

You cannot check if (users[addr].balance > 0) to determine if a user exists — they might legitimately have zero balance. Always use an explicit boolean flag: mapping(address => bool) public isRegistered;. This costs one extra SSTORE (20,000 gas) on registration but prevents logic bugs that are far more expensive.

Interview Question

"What's the gas cost of reading from a nested mapping like mapping(a => mapping(b => uint))?" Answer: Two SLOADs. First SLOAD computes the inner mapping's base slot. Second SLOAD reads the actual value. Cost: 2,100 + 100 = 2,200 gas (second is warm due to same transaction). If both are cold: 4,200 gas.

Function Visibility & Modifiers: The First Line of Defense

Learning Objective

Internalize the four visibility levels, understand when and why to use each, and learn to write modifiers that enforce security invariants without introducing reentrancy or gas waste.

Why external Beats public for External-Only Functions

A public function generates two code paths: one for external calls (reads calldata) and one for internal calls (reads from stack/memory). If a function is only ever called externally, external saves deployment bytecode size and avoids an unnecessary internal dispatch jump. For functions called frequently (like in DeFi), this matters. Rule: If it's never called internally, make it external.

Modifiers: Power and Peril

Modifiers inject code before and after a function body. They're powerful for access control but dangerous if misused. Here's the pattern every production contract uses:

// ✅ GOOD: Simple, single-purpose modifier
modifier onlyOwner() {
    require(msg.sender == owner, "Not owner");
    _;
}

// ❌ BAD: Modifier with state changes (anti-pattern)
modifier deductFee(uint256 amount) {
    uint256 fee = amount / 100;
    balances[msg.sender] -= fee; // ⛔ State change BEFORE _
    _;
    // If _ reverts, the fee deduction is rolled back — but the intent is unclear
}

// ✅ BETTER: Keep modifiers for checks, functions for actions
modifier nonZeroAddress(address _addr) {
    require(_addr != address(0), "Zero address");
    _;
}

// ✅ PRODUCTION: OpenZeppelin's Ownable pattern (simplified)
modifier onlyOwner() {
    require(msg.sender == _owner, "Unauthorized");
    _;
}
The Modifier Reentrancy Trap

If a modifier performs a state change before _; and the function body makes an external call, an attacker can reenter. Always: checks → effects → interactions. Modifiers should only contain checks. Effects and interactions belong in the function body where the ordering is visible.

Lesson 2.3~60 min

Error Handling: Custom Errors vs. Revert Strings

Learning Objective

Understand why require("string") is a gas leak, master custom errors, and learn when to use revert(), assert(), and require() correctly.

// ❌ EXPENSIVE: Revert strings cost gas to store in bytecode
require(balance >= amount, "Insufficient balance");
// Gas: ~2,400+ for the string storage + comparison

// ✅ CHEAP: Custom errors — ~90% less gas for the revert path
error InsufficientBalance(uint256 available, uint256 required);
require(balance >= amount);
// Better: if (balance < amount) revert InsufficientBalance(balance, amount);

// ✅ assert() — for invariants that should NEVER be false
assert(totalSupply >= userBalance); // If this fails, the system is broken

// ✅ revert() — for unconditional reverts
if (block.timestamp > deadline) revert Expired();
The Gas Math That Makes Custom Errors Mandatory

A revert string "Insufficient balance" is 21 bytes. At deployment, each byte of string literal costs ~3 gas for bytecode storage. But the real cost is at runtime: PUSHn opcodes to load the string, then a comparison. A custom error with parameters costs only the selector (4 bytes, always) plus parameter encoding. For a function that reverts frequently (like a transfer that fails 30% of the time), switching to custom errors can save thousands of gas per failed transaction across your user base. At protocol scale, this is millions of dollars.

Pro Insight: Error Selectors Are First-Class Data

Custom errors have a 4-byte selector just like function selectors. You can compute it: keccak256("InsufficientBalance(uint256,uint256)") → first 4 bytes. Frontends can decode this selector to show user-friendly messages. Libraries like viem do this automatically. This is why custom errors are strictly superior to strings in every dimension: cheaper, typed, decodable, and ABI-compatible.

Abstract Contracts & Interfaces: Building Modular Systems

Learning Objective

Distinguish between interfaces and abstract contracts, understand when each is appropriate, and learn how protocols like Uniswap and Aave use these patterns to build upgradeable, composable systems.

Interface vs Abstract — The Real Difference

PropertyInterfaceAbstract Contract
State variablesNoneAllowed
Implemented functionsNoneAllowed
ConstructorsNoneAllowed
ModifiersNoneAllowed
Use caseCross-contract communicationShared base logic with extensions
 
// Interface: The contract boundary — what the outside world sees
interface IERC20 {
    function totalSupply() external view returns (uint256);
    function balanceOf(address account) external view returns (uint256);
    function transfer(address to, uint256 amount) external returns (bool);
    function approve(address spender, uint256 amount) external returns (bool);
    function transferFrom(address from, address to, uint256 amount) external returns (bool);
}

// Abstract: Shared logic that child contracts extend
abstract contract Ownable {
    address internal _owner;

    event OwnershipTransferred(address indexed prev, address indexed next);

    constructor() { _owner = msg.sender; emit OwnershipTransferred(address(0), msg.sender); }

    modifier onlyOwner() { require(msg.sender == _owner); _; }

    function transferOwnership(address newOwner) external onlyOwner {
        require(newOwner != address(0));
        emit OwnershipTransferred(_owner, newOwner);
        _owner = newOwner;
    }
}
Why Uniswap Uses Interfaces for Everything

When you call IERC20(token).transferFrom(), Solidity compiles this to a single CALL opcode with a 4-byte function selector. No contract code is deployed for the interface — it's purely a type-safe ABI descriptor. This means zero deployment gas cost for interfaces, and any contract that implements the IERC20 functions can be called through it. This is composability in its purest form.

Libraries & "Diamond" Standards: Reusing Audited Code

Learning Objective

Understand how Solidity libraries work at the bytecode level, when to use them vs inline code, and why protocols like OpenZeppelin structure their code as libraries.

How Libraries Actually Work

A library is a contract that cannot receive ETH, cannot have state variables, and cannot be destroyed. When you call a library function, the compiler either: (a) inlines the library code directly into your contract (for internal/pure/view functions) or (b) makes a DELEGATECALL to the deployed library (for functions that modify state). Option (a) costs no extra gas. Option (b) costs an extra CALL (~2,600 gas) but saves deployment gas if multiple contracts share the library.

// Library: Inlined (internal) — zero extra runtime gas
library SafeMath {
    function add(uint256 a, uint256 b) internal pure returns (uint256) {
        uint256 c = a + b;

DELEGATECALL Storage Danger

When a library uses DELEGATECALL, it executes in the calling contract's storage context. If the library expects state at slot 0 and your contract has different data at slot 0, the library writes to the wrong slot. This caused the Parity Wallet hack. Only use DELEGATECALL libraries with using ... for syntax, which handles storage layout automatically.

Reentrancy: The Classic DAO Hack

Learning Objective

Understand reentrancy at the EVM level, implement a vulnerable contract, write an exploit, then fix it. This is the most important security lesson in the entire course.

How Reentrancy Works — Step by Step

// The vulnerable pattern:
function withdraw() external {
    uint256 amount = balances[msg.sender]; // 1. Read balance
    (bool ok,) = msg.sender.call{value: amount}(""); // 2. Send ETH (CALL — can trigger fallback)
    require(ok); // 3. Check success
    balances[msg.sender] = 0; // 4. ZERO balance ← TOO LATE!
}

// Attacker's fallback function:
receive() external payable {
    if (gasleft() > 20000) {
        victim.withdraw(); // Re-enter before step 4!
    }
}

// Execution flow:
// withdraw() → call(attacker) → attacker.fallback() → withdraw() → call(attacker) → ...
// Each reentry reads the SAME balance (not yet zeroed) → drains entire contract
// ✅ FIX 1: Checks-Effects-Interactions Pattern
function withdraw() external {
    uint256 amount = balances[msg.sender];
    require(amount > 0);                    // CHECK
    balances[msg.sender] = 0;                 // EFFECT (zero BEFORE external call)
    (bool ok,) = msg.sender.call{value: amount}(""); // INTERACTION (last)
    require(ok);
}

// ✅ FIX 2: ReentrancyGuard (OpenZeppelin)
modifier nonReentrant() {
    require(!_locked, "ReentrancyGuard: reentrant call");
    _locked = true;
    _;
    _locked = false;
}

// ✅ FIX 3: Pull over Push (user calls claim, not withdraw)
// Instead of contract pushing ETH, user pulls it:
function claim() external {
    uint256 amount = pendingWithdrawals[msg.sender];
    pendingWithdrawals[msg.sender] = 0;
    msg.sender.call{value: amount}("");
}
Cross-Function Reentrancy: The Sneaky Variant

Reentrancy doesn't have to re-enter the same function. If function A calls an external contract, and that contract calls function B which shares state with A, that's cross-function reentrancy. The CEI pattern protects against this too — but a nonReentrant guard on only function A does NOT. Either use nonReentrant on ALL external-facing state-changing functions, or rely on CEI ordering exclusively.

Integer Overflows & Underflows: Life After Solidity 0.8

Solidity 0.8+ has built-in overflow checks. So this lesson is short — but critical. Because the danger hasn't disappeared; it's moved to two places: (1) pre-0.8 contracts you'll audit, and (2) unchecked blocks you'll write for gas optimization.

// Pre-0.8: uint256 max + 1 = 0 (wraps around)
// 0.8+: uint256 max + 1 = REVERT (automatic check)

// ⛔ DANGER: unchecked block disables the safety net
function dangerousSubtraction(uint256 a, uint256 b) internal pure returns (uint256) {
    unchecked {
        return a - b; // If b > a, this wraps to a huge number!
    }
}

// ✅ SAFE: Only use unchecked when you PROVE it can't overflow
function safeIncrement(uint256 x) internal pure returns (uint256) {
    unchecked {
        return x + 1; // Safe: uint256 max is 2^256-1, impossible to reach
    }
}

// Gas savings of unchecked: ~30 gas per arithmetic operation
// Only use in hot loops where the savings compound

"When is it safe to use unchecked?" Answer: Only when you can mathematically prove the operation cannot overflow/underflow. Common safe patterns: counter increments in bounded loops, operations where a prior require() guarantees the range, and i++ in for loops where the upper bound is checked. If you can't write the proof, don't use unchecked.

Front-running & MEV: How Bots "Eat" Your Users' Profit

Learning Objective

Understand how transactions in the mempool are visible to MEV searchers, the three primary attack types (front-running, back-running, sandwich), and the mitigations available at the contract level.

Mental Model: The Insider Trading Analogy

Imagine you're at a stock exchange and you can see everyone's orders before they execute. If someone places a massive buy order for Apple stock, you can buy Apple first (front-run), sell to them at the higher price they created (back-run), or buy before AND sell after (sandwich). That's exactly what MEV bots do on Ethereum — except it's technically legal because the mempool is public by design.

Front-Running

Bot sees your tx in the mempool, copies your calldata with a higher gas price, gets included before you. Example: you submit a swap on a DEX, bot swaps first at the better price, you get a worse rate.

// Mempool order:
Bot tx      (gas: 50 gwei) → swap 100 ETH for USDC
Your tx      (gas: 30 gwei) → swap 100 ETH for USDC
// Bot gets better price. You get worse price. Bot profits the difference.
Sandwich Attack

Most profitable MEV pattern. Bot places a tx BEFORE yours (front-run) and a tx AFTER yours (back-run). You get sandwiched between two trades that extract your slippage as profit.

Bot: buy token X       (pushes price up)
You: buy token X       (at artificially high price)
Bot: sell token X      (profits from the price increase you caused)
// You lose ~0.5-2% per trade. Bot makes ~$1M/day on Uniswap alone.
Back-Running

Bot sees your tx execute (usually via the pending block on a builder), then places a tx immediately after to profit from the state change you created. Example: your liquidation triggers a price move, bot arbitrages that move.

// ❌ VULNERABLE: Price changes visible before execution
function swap(uint256 amountIn) external {
    uint256 amountOut = calculateOutput(amountIn);
    // ↑ Price calculated from current reserve (visible in mempool)
    // ↓ By the time this executes, a bot has already swapped
    _transfer(msg.sender, amountOut);
}

// ✅ MITIGATION 1: Commit-Reveal pattern
function commitSwap(bytes32 hash) external {
    commits[msg.sender] = hash; // Hash of (amount, secret)
}
function revealSwap(uint256 amount, bytes32 secret) external {
    require(keccak256(abi.encode(amount, secret)) == commits[msg.sender]);
    // Now execute — amount was hidden during commit phase
}

// ✅ MITIGATION 2: Private mempools (Flashbots Protect, MEV Blocker)
// Not a contract change — users route txs through private relays
// Bots cannot see the tx until it's in the block

// ✅ MITIGATION 3: Slippage protection (frontend-level, but critical)
// minAmountOut parameter — revert if output < minimum acceptable
function swap(uint256 amountIn, uint256 minAmountOut) external {
    uint256 amountOut = calculateOutput(amountIn);
    require(amountOut >= minAmountOut, "Slippage exceeded");
    // If a bot front-runs and pushes price, amountOut < min → revert
}
Pro Insight: MEV Is Not a Bug — It's a Feature of Transparent Blockchains

You cannot eliminate MEV at the contract level. You can only: (1) make it unprofitable for bots (tight slippage), (2) hide information before execution (commit-reveal), or (3) route through private mempools. As a contract engineer, your job is to ensure your protocol doesn't create MEV opportunities through design flaws — like oracle updates that can be front-run (covered in Chapter 9).

How This Fails in Production: The $6M Balancer Exploit (2023)

Balancer's vault had a function where the fee percentage could be changed, and the change took effect in the same block. An attacker saw the pending tx, front-ran it with a swap that profited from the old fee rate, then the fee changed. Arithmetic & Logical Errors: The Silent Killers

Learning Objective

// ❌ SILENT BUG: Integer division truncation
function calculateFee(uint256 amount) public pure returns (uint256) {
    return amount * 3 / 100; // 3% fee
    // amount = 1 → 1 * 3 / 100 = 0 (fee is ZERO!)
    // amount = 33 → 33 * 3 / 100 = 0 (fee is ZERO!)
    // amount = 34 → 34 * 3 / 100 = 1 (should be 1.02)
}

// ✅ FIX: Multiply before divide (already doing it above, but...)
// The real fix is ensuring amount is never small enough to truncate to 0

// ❌ SILENT BUG: Wrong order in fee sharing
function distribute(uint256 amount) external {
    uint256 protocolFee = amount / 100;       // 1%
    uint256 lpFee = (amount - protocolFee) / 10; // 9.9% of original?
    // NO! lpFee = (amount - amount/100) / 10
    // amount=1000: protocolFee=10, lpFee=99. Expected: 990/10=99. Actual: 99. OK here.
    // But total distributed = 10 + 99 = 109, not 110. 1 wei "leaks" to no one.
}

// ✅ FIX: Calculate all fees from the ORIGINAL amount
function distribute(uint256 amount) external {
    uint256 protocolFee = amount / 100;
    uint256 lpFee = amount / 10; // Calculate from original, not residual
    uint256 userAmount = amount - protocolFee - lpFee;
    // Now: 1000 → 10 + 100 + 890 = 1000. No leakage.
}

// ❌ SILENT BUG: Rounding in the wrong direction
// When calculating user payouts, rounding DOWN favors the protocol.
// When calculating protocol fees, rounding DOWN favors the user.
// Always round in the direction that is LESS favorable to YOU
// to avoid slowly draining the protocol over thousands of txs.
The "Death by a Thousand Cuts" Gas-Arithmetic Tradeoff

Some protocols use mulDiv from OpenZeppelin (256-bit full-precision multiplication/division) to avoid truncation. It costs ~200 extra gas per call vs native division. At 1M txs/day, that's 200M gas/day = ~$60K/day at 30 gwei. You must decide: is the truncation loss greater or less than the extra gas cost? For fee calculations, almost always use mulDiv. For display formatting, native division is fine.

Fuzzing: Letting the Computer Find Your Bugs

Learning Objective

Understand property-based testing, write Foundry fuzz tests that explore input spaces you'd never think of manually, and learn to guide the fuzzer with assumptions and invariant assertions.

Mental Model: The Robot That Tests While You Sleep

A unit test checks one input: transfer(bob, 100). A fuzz test checks MILLIONS of inputs: transfer(bob, 0), transfer(bob, 1), transfer(bob, 2^256-1), transfer(bob, type(uint256).max/2), transfer(zeroAddress, 999)... It finds edge cases your brain would never conceive. Foundry runs 256 random inputs per fuzz test by default, but you can configure it to run millions.

// UNIT TEST: Checks ONE specific input
function test_Transfer() public {
    token.transfer(bob, 100);
    assertEq(token.balanceOf(bob), 100);
}

// FUZZ TEST: Checks MILLIONS of inputs automatically
function testFuzz_Transfer(uint256 amount) public {
    vm.assume(amount <= token.balanceOf(address(this))); // Filter invalid inputs
    vm.assume(amount > 0); // Filter trivial inputs
    vm.assume(bob != address(0)); // Filter invalid recipients

    uint256 balanceBefore = token.balanceOf(bob);
    token.transfer(bob, amount);
    assertEq(token.balanceOf(bob), balanceBefore + amount);
}

// Run with: forge test --fuzz-runs 10000
// Foundry tries 10,000 random uint256 values for 'amount'
// Any failure = the test FAILS with the specific input that broke it

// ADVANCED: Fuzzing structured inputs
function testFuzz_Swap(
    address tokenIn,
    address tokenOut,
    uint256 amountIn
) public {
    vm.assume(tokenIn != tokenOut);
    vm.assume(tokenIn != address(0));
    vm.assume(amountIn > 0 && amountIn < 1e30);

    // The fuzzer will find combinations you never thought of:
    // - tokenIn = address(this) (the contract itself)
    // - amountIn = type(uint256).max (overflow scenarios)
    // - tokenIn = address(pool) (self-interaction)
}
Fuzz Run Count vs. Gas Cost Tradeoff

Default: 256 runs. For critical functions: 10,000+. For public mainnet contracts: 100,000+. Running forge test --fuzz-runs 100000 on a complex function can take 5–10 minutes and cost ~$0.50 in compute. That $0.50 has saved protocols tens of millions of dollars. It's the highest-ROI gas you'll ever spend.

The vm.assume() Trap

vm.assume() does NOT reduce the input space — it rejects invalid inputs and retries. If 99.9% of random inputs are invalid, Foundry will waste 99.9% of its runs on rejected inputs and barely explore the valid space. If your assume() rejects more than ~90% of inputs, restructure your test or use boundInt from forge-std instead.

Invariant Testing: Defining Rules That Can Never Break

Learning Objective

This is the most powerful testing technique in Foundry. Invariant testing lets you define rules like "totalSupply always equals sum of all balances" and then Foundry generates random sequences of function calls trying to break that rule. It's how OpenZeppelin tests their own libraries.

Mental Model: The Chaos Monkey

Imagine a monkey that randomly presses buttons on your contract: deposit, withdraw, transfer, approve, transferFrom — in random order, with random amounts, from random addresses, 10,000 times per second. An invariant is a rule you declare: "no matter what that monkey does, this must ALWAYS be true." If the monkey finds a sequence that breaks your rule, Foundry tells you exactly which sequence did it.

// Handler contract: defines what the "monkey" can do
contract TokenHandler is Test {
    Token token;
    address[] public users;

    function setUp() public {
        token = new Token(1_000_000e18);
        users = [alice, bob, carol];
        for (uint i; i < users.length; i++) {
            token.transfer(users[i], 1000e18);
        }
    }

    // The monkey can call this with ANY amount and ANY user
    function transfer_random(uint256 senderIdx, uint256 receiverIdx, uint256 amount) public {
        address sender = users[senderIdx % users.length];
        address receiver = users[receiverIdx % users.length];
        vm.prank(sender);
        token.transfer(receiver, amount);
    }
}

// Invariant test: defines the UNBREAKABLE rules
contract TokenInvariantTest is Test {
    TokenHandler handler;

    function setUp() public {
        handler = new TokenHandler();
        targetContract(address(handler)); // Tell Foundry where to send fuzz calls
    }

    // INVARIANT 1: Total supply never changes (no mint/burn)
    function invariant_totalSupply_NeverChanges() public {
        assertEq(handler.token().totalSupply(), 1_000_000e18);
    }

    // INVARIANT 2: Sum of all balances == totalSupply
    function invariant_sumBalances_EqualsTotalSupply() public {
        uint256 sum;
        for (uint i; i < handler.users().length; i++) {
            sum += handler.token().balanceOf(handler.users()[i]);
        }
        sum += handler.token().balanceOf(address(handler));
        assertEq(sum, handler.token().totalSupply());
    }

    // INVARIANT 3: No user can have more than totalSupply
    function invariant_noUserExceedsTotalSupply() public {
        for (uint i; i < handler.users().length; i++) {
            assertLe(handler.token().balanceOf(handler.users()[i]),
                       handler.token().totalSupply());
        }
    }
}

# Run: forge test -- invariant -- invariant-runs 10000
# Foundry generates 10,000 random CALL SEQUENCES and checks invariants after each
How to Write Good Invariants

Bad invariant: "balanceOf(alice) >= 0" — this is always true by type system, wastes a run. Good invariant: "sum(allBalances) == totalSupply" — this actually tests something. For every contract, ask: "What property, if violated, means the contract is broken?" That's your invariant. Common patterns: conservation (sum in == sum out), monotonicity (counter only increases), bounds (value between min and max).

The "Handler State" Footgun

Your handler contract has its own state. If the handler calls token.transfer(alice, amount) and the transfer reverts, the handler's state might be inconsistent. Always use try/catch in handlers for functions that can revert, and never assume a call succeeded. A reverted call in a handler sequence should be a no-op, not a state corruption.

Proxy Patterns: Transparent vs. UUPS

Learning Objective

Understand WHY proxies exist, HOW DELEGATECALL makes them work at the EVM level, and the critical differences between Transparent and UUPS proxy patterns — including why the industry moved from one to the other.

Mental Model: The Restaurant That Changed Its Kitchen

Imagine a restaurant at a fixed address. Customers always go to that address to eat. But the kitchen behind it gets replaced every few months — better equipment, new menu, fixed health code violations. The address (proxy) never changes. The kitchen (implementation) gets swapped. Customers don't need to know. That's what a proxy does: the contract address stays the same, but the code behind it can be upgraded.

The mechanism is DELEGATECALL. When the proxy receives a call, it doesn't execute any logic itself. It forwards the ENTIRE call (calldata, msg.sender, msg.value) to the implementation contract. The implementation executes in the proxy's storage context. The result is returned to the caller as if the proxy executed it.

// TEXT DIAGRAM: How a Proxy Call Works

User
  │ "transfer(bob, 100)"
  ▼
Proxy Contract (address: 0xAAA... — this is what users know)
  │ fallback() {
  │   (bool ok,) = implementation.delegatecall(msg.data);
  │   assembly { return(0, returndata_size) }
  │ }
  ▼ DELEGATECALL (executes in proxy's storage, not implementation's)
Implementation V1 (address: 0xBBB... — nobody talks to this directly)
  │ function transfer(address to, uint256 amount) {
  │   balances[msg.sender] -= amount;  ← writes to PROXY's slot 0
  │   balances[to] += amount;           ← writes to PROXY's slot 1
  │ }
  ▼ return data flows back through proxy to user
User gets response — thinks 0xAAA executed it

// To upgrade: admin calls proxy.upgradeTo(0xCCC...) — new implementation address
// All storage at 0xAAA is preserved. Only the code changes.

If you deploy UUPS V2 and accidentally remove the upgradeTo() function, the proxy is permanently stuck on V2. No more upgrades. Ever. This happened to a real protocol. Mitigation: your UUPS implementation MUST inherit from ERC1967Upgrade and always include the upgrade function. Write a test that calls upgradeTo and assert it succeeds before every deployment.

Storage Collisions: The Danger of Changing Variables

Learning Objective

Understand exactly how proxy storage layout works, why adding/removing/reordering variables in an upgrade causes silent data corruption, and how to use forge inspect storageLayout to verify safety before upgrading.

// THE #1 UPGRADE BUG: Storage Layout Mismatch
//
// V1 Implementation:
contract MyTokenV1 {
    uint256 public totalSupply;   // Proxy slot 0
    address public owner;        // Proxy slot 1
    mapping(address => uint256) public balances; // Proxy slot 2 (mapping seed)
}

// V2 Implementation (DANGEROUS — variable added in middle!):
contract MyTokenV2 {
    uint256 public totalSupply;   // Still slot 0 ✓
    bool    public paused;       // ⛔ NEW — now slot 1!
    address public owner;        // ⛔ NOW slot 2 — reads PAUSED data as ADDRESS!
    mapping(address => uint256) public balances; // ⛔ NOW slot 3 — all balances LOST!
}

// After upgrade:
// owner = keccak of paused bool value (garbage address)
// balances = reads from slot 3, but data is in slot 2 = ALL BALANCES ZERO
// Users see 0 balance. Total supply unchanged. Silent corruption.
// ✅ SAFE V2: New variables appended at the END only
contract MyTokenV2 {
    uint256 public totalSupply;   // Slot 0 — unchanged
    address public owner;        // Slot 1 — unchanged
    mapping(address => uint256) public balances; // Slot 2 — unchanged
    // ─── NEW VARIABLES BELOW THIS LINE ONLY ───
    bool    public paused;       // Slot 3 — new, no collision
}

# Verify before upgrading:
forge inspect MyTokenV1 storageLayout > v1_layout.json
forge inspect MyTokenV2 storageLayout > v2_layout.json
# Compare: slots 0-2 must be identical types and names
# Slot 3+ can be new additions only
How This Fails in Production: The Parity Wallet Freeze

In 2017, someone called initWallet() on a Parity multisig library contract, which overwrote the owner at slot 0. Later, the same person called kill() and destroyed the library. Since all Parity wallets used DELEGATECALL to this library, $150M in ETH was permanently frozen because the fallback function now pointed to a self-destructed contract. This is the most expensive storage collision in history.

The "EIP-1967 Storage Slots" Safety Net

OpenZeppelin's proxy contracts store the implementation address and admin at fixed, well-known storage slots far from the normal variable range: bytes32(uint256(keccak256("eip1967.proxy.implementation")) - 1). This prevents the implementation from accidentally overwriting proxy metadata. Always use OZ proxies — never write your own delegatecall logic.

Initializers vs. Constructors

In a proxy pattern, the constructor runs on the IMPLEMENTATION contract at deployment — not on the proxy. This means constructor-set state lives in the implementation, not the proxy. Users interact with the proxy, which never ran the constructor. This is why initialize() exists.

// ❌ WRONG: Constructor state lives in implementation, not proxy
contract MyTokenV1 {
    address public owner;
    constructor() {
        owner = msg.sender; // Runs at impl deployment (deployer = owner)
        // Proxy's slot 0 is EMPTY — owner was set in impl's storage!
    }
}

// ✅ CORRECT: Initializer runs on the proxy via delegatecall
import "@openzeppelin/contracts-upgradeable/proxy/utils/Initializable.sol";

AMM Math: The Constant Product Formula (x·y = k)

Learning Objective

Understand the invariant-based design of constant product AMMs, derive the swap formula from first principles, and implement the core math in Solidity. This mechanism eliminates order book dependencies while maintaining continuous liquidity availability.

System Model: Invariant-Based Liquidity Pools

Traditional exchanges require order books: matching buy and sell orders at crossing prices. AMMs replace this with a mathematical invariant. The pool holds both tokens in a ratio determined by the constant product (x·y = k). When a swap occurs, the ratio shifts—this shift IS the execution cost. The invariant ensures the pool never reaches zero liquidity: larger swaps incur proportionally higher execution costs, creating a self-regulating mechanism against pool depletion.

// Pool state:
Reserve ETH: 100    ← x
Reserve USDC: 200,000  ← y
k = x × y = 20,000,000  ← INVARIANT (before fees)

// Exchange rate = y / x = 200,000 / 100 = 2,000 USDC per ETH

// Swap 1 ETH in:
// New x = 101, new y = k / 101 = 198,019.8
// Output: 200,000 - 198,019.8 = 1,980.2 USDC
// (Not 2,000 — execution cost due to slippage)
// This execution cost compensates liquidity providers for
// maintaining the pool's availability
// Core swap math — derived from x*y = k invariant
function getAmountOut(
    uint256 amountIn,
    uint256 reserveIn,
    uint256 reserveOut
) internal pure returns (uint256 amountOut) {
    require(amountIn > 0, "Insufficient input");
    require(reserveIn > 0 && reserveOut > 0, "Insufficient liquidity");

    uint256 fee = amountIn * 3 / 1000; // 0.3% infrastructure fee
    uint256 amountInWithFee = amountIn - fee;

    // Derived from: (reserveIn + amountInWithFee) * (reserveOut - amountOut) = reserveIn * reserveOut
    // Solving for amountOut:
    amountOut = (amountInWithFee * reserveOut) / (reserveIn + amountInWithFee);
}

// The fee is extracted BEFORE the swap calculation
// This causes k to INCREASE with each swap — the delta
// accumulates as system reserves for liquidity providers
// k_before = 100 * 200000 = 20,000,000
// k_after  = 101 * 198019.8 + accumulated_fee
// The fee accumulation compensates for pool state drift
Execution Costs & Liquidity Infrastructure Fees

The 0.3% fee is a distributed system service fee—it covers the cost of maintaining continuous liquidity availability and compensates for the state drift that occurs with each swap. For high-volatility token pairs, higher fees offset increased drift magnitude. For stable pairs (USDC/USDT), minimal drift occurs, so lower fee tiers (0.01%, 0.05%) maintain system efficiency. When designing an AMM, fee tiers must match the pair's volatility profile to maintain system equilibrium.

Liquidity Provision: State Representation via LP Tokens

LP tokens are on-chain state representations that track a user's proportional claim on pool reserves. They serve as internal accounting mechanisms for liquidity positions—minted on deposit, burned on withdrawal, and transferable between addresses.

// LP tokens represent proportional state claims on the pool
// Minted on deposit, burned on withdrawal
// Total LP supply = pool's total tracked size

function addLiquidity(
    uint256 amountToken0,
    uint256 amountToken1
) external returns (uint256 liquidity) {
    // First deposit: LP tokens = sqrt(amount0 * amount1)
    if (totalSupply == 0) {
        liquidity = Math.sqrt(amountToken0 * amountToken1);
    } else {
        // Subsequent: mint proportional to min ratio
        uint256 liquidity0 = (amountToken0 * totalSupply) / reserve0;
        uint256 liquidity1 = (amountToken1 * totalSupply) / reserve1;
        liquidity = Math.min(liquidity0, liquidity1);
    }
    // ⛔ IMPORTANT: Only deposit the MINIMUM ratio to avoid
    // depositing excess tokens that don't track state changes
    // (this is the "dust" problem — untracked residual tokens)
    _mint(msg.sender, liquidity);
}

function removeLiquidity(
    uint256 liquidity
) external returns (uint256 amount0, uint256 amount1) {
    amount0 = (liquidity * reserve0) / totalSupply;
    amount1 = (liquidity * reserve1) / totalSupply;
    _burn(msg.sender, liquidity);
    // Transfer amount0 and amount1 to user
}

// KEY INSIGHT: LP tokens are a CLAIM on a FRACTION of the pool,
// not on fixed amounts. If swaps change the reserve ratio,
// LP tokens represent different amounts of each token,
// but always the same FRACTION of total pool state.
The sqrt() Initial Mint: Overflow Failure Mode

The first LP deposit uses sqrt(amount0 * amount1) to set the initial LP token count. This MUST use a safe sqrt implementation that handles overflow. If amount0 * amount1 overflows uint256 (which occurs at ~1.3e38 for each), the entire pool initialization fails or produces incorrect state. Use OpenZeppelin's Math.sqrt() or the Babylonian method. Never implement custom square root without formal verification.

Atomic Liquidity & Zero-Capital Execution Systems

System Model: Atomic Resource Access

Atomic liquidity systems allow temporary access to pool reserves within a single transaction—zero upfront capital required. The constraint: borrowed resources must be returned (plus execution fee) within the same atomic transaction. If the callback fails to restore the pool state, the entire transaction reverts. This mechanism enables efficient capital usage for system-level operations: cross-pool price correction, portfolio rebalancing, and collateral swaps—all executing atomically without requiring external capital lockup.

function flashSwap(
    uint256 amount0Out,
    uint256 amount1Out,
    address to,
    bytes calldata data
) external {
    // Step 1: Transfer tokens to caller (NO collateral required)
    if (amount0Out > 0) _safeTransfer(token0, to, amount0Out);
    if (amount1Out > 0) _safeTransfer(token1, to, amount1Out);

    // Step 2: Callback — caller executes arbitrary logic
    if (data.length > 0) {
        IUniswapV2Callee(to).uniswapV2Call(msg.sender, amount0Out, amount1Out, data);
    }

    // Step 3: Verify pool state — caller MUST have restored balance + fee
    uint256 balance0 = IERC20(token0).balanceOf(address(this));
    uint256 balance1 = IERC20(token1).balanceOf(address(this));
    require(balance0 >= reserve0 + amount0Out * 1000 / 997, "Pool state not restored");
    // 1000/997 = 1.003... → 0.3% execution fee on flash operations
    // If require fails → ENTIRE tx reverts → pool state preserved
}

function uniswapV2Call(address sender, uint256 amount0, uint256 amount1, bytes calldata) external {

    // 2. Execute swap on pool B: 1000 USDC → 0.51 ETH
    // 3. Return 1003 USDC to pool A (1000 + 0.3% fee)
    // 4. Result: Pool A/B prices converge, system efficiency improved
}
Atomicity Guarantees Pool Integrity

Tokens are transferred out in step 1. Step 3 verifies the post-callback balance meets the invariant. If the caller fails to restore state, the require fails and the ENTIRE transaction reverts—including the initial transfers. EVM atomicity makes it mathematically impossible to drain a pool through flash operations. The system is invariant-safe by design.

When building systems that interact with flash-capable pools, treat every external call as a potential attack vector. Incentive Distribution: O(1) Staking Reward Systems

Many distributed systems require mechanisms to incentivize participation—liquidity provision, validation, or resource contribution. The core engineering challenge is distributing rewards across potentially thousands of participants with constant gas cost per interaction. The Synthetix pattern solves this through lazy evaluation of accumulated rewards.

// Incentive distribution: stake tokens, receive reward tokens over time
// Core challenge: O(1) gas per user regardless of total participants
contract IncentivePool {
    IERC20 public stakingToken;    // Token being staked (e.g., LP token)
    IERC20 public rewardToken;     // Token distributed as incentive
    uint256 public rewardRate;        // Rewards per second
    uint256 public lastUpdateTime;
    uint256 public rewardPerTokenStored; // Global accumulated reward per staked token
    mapping(address => uint256) public userRewardPerTokenPaid;
    mapping(address => uint256) public rewards;

    function rewardPerToken() public view returns (uint256) {
        if (totalStaked == 0) return rewardPerTokenStored;
        return rewardPerTokenStored +
            (block.timestamp - lastUpdateTime) * rewardRate * 1e18 / totalStaked;
    }

    function earned(address account) public view returns (uint256) {
        return (balanceOf(account) * (rewardPerToken() - userRewardPerTokenPaid[account]))
            / 1e18 + rewards[account];
    }

    function stake(uint256 amount) external updateReward(msg.sender) {
        // updateReward computes global reward delta before any balance change
        _transfer(msg.sender, address(this), amount);
    }

    // This pattern scales to thousands of participants with
    // O(1) gas per deposit/withdraw/claim — no iteration over participants
}
O(1) vs O(N) Distribution — Scalability Requirement

Naive approach: iterate through all participants each block, increment each reward. Gas: O(N)—fails at 100+ participants. Synthetix pattern: store global accumulated reward per token, each participant computes their share lazily on interaction. Gas: O(1) per participant regardless of total count. This pattern is foundational for any system requiring proportional distribution across unbounded participant sets.

Collateralized Risk Management & Liquidation Systems

System Model: Overcollateralized Asset Positions

Collateralized finance systems require users to lock assets exceeding their position value—this overcollateralization is the primary defense against system insolvency. If collateral value drops relative to the position, it becomes undercollateralized and must be liquidated: the collateral is sold to cover the position, and a liquidation incentive ensures external actors perform this maintenance. This mechanism prevents bad position accumulation and maintains system solvency under market stress.

// Health Factor = (Collateral Value × LTV) / Position Value
// LTV = Loan-to-Value ratio (max allowed position as fraction of collateral)
//
// Example:
// Collateral: 10 ETH × $2,000 = $20,000
// LTV: 75% (maximum position threshold)
// Max position size: $20,000 × 0.75 = $15,000
// Actual position: $10,000
// Health Factor = ($20,000 × 0.75) / $10,000 = 1.5
//
// Market stress — ETH drops to $1,200:
// Collateral: 10 ETH × $1,200 = $12,000
// Health Factor = ($12,000 × 0.75) / $10,000 = 0.9
// ⛔ Health Factor < 1.0 → LIQUIDATION TRIGGERED
// System sells collateral to cover position, preventing insolvency
// Liquidation incentive (5%) compensates the liquidator for execution
function healthFactor(address user) public view returns (uint256) {
    uint256 collateralValue = getCollateralValue(user);
    uint256 positionValue = getPositionValue(user);
    if (positionValue == 0) return type(uint256).max;
    return (collateralValue * 1e18 * ltv) / positionValue; // Scaled for precision
}

function liquidate(
    address user,
    uint256 repayAmount
) external {
    require(healthFactor(user) < 1e18, "Position is healthy");

    // Liquidation incentive ensures external actors perform maintenance
    uint256 bonus = repayAmount * liquidationBonus / 100; // e.g., 5%
    uint256 collateralToSeize = (repayAmount + bonus)
        * getPrice(collateralToken) / getPrice(positionToken);

    // Position reduction and collateral transfer
    positions[user] -= repayAmount;
    collateral[user] -= collateralToSeize;
    collateral[msg.sender] += collateralToSeize;
}

// ⛔ CRITICAL: Prices MUST come from verified external data feeds
// Using unverified on-chain prices enables manipulation → system failure

An actor can transfer tokens directly to a user's address—tokens the protocol doesn't track in internal state. While this doesn't directly drain funds, it can cause withdrawal failures if the protocol compares balanceOf() against internal accounting. Dynamic Return Models: Performance-Linked Capital Allocation

Performance-linked return models serve as feedback control systems that maintain pool equilibrium. By dynamically adjusting return allocation based on utilization and actual pool performance, these mechanisms ensure fair distribution between capital providers and system operators while preventing resource exhaustion. Unlike fixed-cost models where returns are guaranteed regardless of outcome, performance-linked systems tie compensation to real results—aligning incentives across all participants.

Economic Paradigm Comparison

// ════════════════════════════════════════════════════════
// PARADIGM A: FIXED-COST MODEL (Guaranteed Returns)
// ────────────────────────────────────────────────────
// Characteristic: Capital provider receives predetermined %
// Risk Profile: Provider bears default risk only
// Return Structure: Fixed/Guaranteed (if no default)
// Example: deposit $1000 → earn 5% annually regardless of pool performance
//
// ════════════════════════════════════════════════════════
// PARADIGM B: PERFORMANCE-LINKED MODEL (Variable Returns)
// ────────────────────────────────────────────────────
// Characteristic: Returns tied to actual pool revenue/performance
// Risk Profile: Shared across ALL participants proportionally
// Return Structure: Variable (no guarantees, based on results)
// Example: invest $1000 → share in profits OR losses based on pool activity
//
// ENGINEERING NOTE:
// Both paradigms require dynamic pricing mechanics to maintain equilibrium.
// We implement Paradigm B as our DEFAULT example (values-aligned).
// Students may choose either paradigm for their final project.

System Model: Utilization-Based Return Allocation

In performance-linked systems, return rates adjust dynamically based on pool utilization—the ratio of active positions to available capital. Low utilization means abundant capital (lower competition for opportunities). High utilization means scarce capital (higher demand, but also higher risk). The system uses a kinked model to gradually increase returns at low utilization, then aggressively increase them past a threshold—discouraging over-utilization that could lead to liquidity crises.

// PERFORMANCE-LINKED RETURN MODEL (Kinked Utilization Curve)
//
// returnRate = baseReturn + utilization × slope1 (below threshold)
// returnRate = rateAtThreshold + (util - threshold) × slope2 (above)
//
// utilization = activePositions / totalCapital
//
// SCENARIO ANALYSIS:
//
// 0% utilization → 2% projected return (base rate, low demand)
// 50% utilization → 2% + 50% × 20% = 12% projected return
// 80% utilization (threshold) → 12% projected return
//
// ABOVE THRESHOLD — Aggressive scaling begins:
// 90% utilization → 12% + 10% × 150% = 27% projected return
// 100% utilization → 42% projected return
//
// WHY THE KINK?
// Below threshold: Gradual increase encourages capital deployment
// Above threshold: Sharp increase discourages over-extension
// Prevents "liquidity run" scenarios where withdrawals fail
//
// KEY DIFFERENCE FROM FIXED-COST MODELS:
// These are PROJECTED returns based on historical performance
// Actual returns depend on REAL pool revenue (not guaranteed)
// If pool loses money → ALL participants share losses proportionally
function getProjectedReturnRate(
    uint256 cash, 
    uint256 activePositions, 
    uint256 reserves
) public view returns (uint256)
{
    uint256 util = activePositions * 1e18 / (cash + activePositions - reserves);

    if (util <= OPTIMAL_UTILIZATION) {
        // Below threshold: gradual return increase
        // Encourages capital deployment when abundance exists
        return BASE_RETURN_RATE + util * SLOPE_1 / 1e18;
    } else {
        // Above threshold: aggressive return increase
        // Discourages over-utilization, prevents exhaustion
        uint256 excessUtil = util - OPTIMAL_UTILIZATION;
        return RATE_AT_THRESHOLD + excessUtil * SLOPE_2 / 1e18;
    }
}

// ACTUAL DISTRIBUTION: Based on REAL pool performance
function distributeReturns(uint256 poolRevenue) external {
    uint256 totalSupply = getTotalSupply();
    
    // Reserve accumulation (protocol sustainability fund)
    uint256 protocolReserve = poolRevenue * reserveFactor / 100; // e.g., 10%
    
    // Distributable amount to capital providers
    uint256 distributable = poolRevenue - protocolReserve;
    
    // Per-unit return calculation (proportional to contribution)

    // Update each provider's accumulated returns
    for (uint256 i = 0; i < providers.length; i++) {
        address provider = providers[i];
        uint256 contribution = contributions[provider];
        accumulatedReturns[provider] += contribution * returnPerUnit / 1e18;
    }
    
    // Note: If poolRevenue is zero or negative (loss),
    // returnPerUnit is zero or negative → shared loss
    // This is RISK SHARING, not guaranteed returns
}

Mental Model: The "I Know a Secret Without Revealing It" Proof

Imagine you want to prove you're over 21 without showing your ID. A ZK proof lets you generate a cryptographic proof that says: "I know a value X such that X > 21, and I can prove I know X without revealing X itself." The verifier can check the proof and be mathematically certain the statement is true — but learns nothing about X. This is the foundational concept.

zk-SNARKs (Succinct Non-Interactive Arguments of Knowledge): Small proof size (~200 bytes), fast verification (~10ms). Used by Zcash, Aztec, Tornado Cash. Requires a trusted setup ceremony.

zk-STARKs (Scalable Transparent Arguments of Knowledge): Larger proofs (~50KB), slower verification, but NO trusted setup. Used by StarkNet, StarkEx (dYdX). More "future-proof."

Application to scaling (zkRollups): Instead of executing 10,000 transactions on-chain, execute them off-chain and submit ONE proof that proves all 10,000 are valid. The chain only verifies the proof (~200 bytes) instead of the transactions. This is how zkSync, Scroll, and Polygon zkEVM work.

// Circom: A DSL for writing ZK circuits
// This circuit proves: "I know x such that x * x == out"

pragma circom 2.1.0;

template SquareProof() {
    // Private input: the value we're proving knowledge of
    signal input x;     // Secret — NOT revealed to verifier
    // Public input: the result that verifier can see
    signal output out;   // Public — verifier knows this value

    // The constraint: x² must equal out
    // If x=5 and out=25, this is satisfied → proof is valid
    // If x=5 and out=26, this is NOT satisfied → proof generation fails
    out === x * x;
}

component main { public [out] = SquareProof(); }

// KEY INSIGHT:
// - The prover runs this circuit with x=5, out=25 → generates proof
// - The verifier gets: proof + out=25 (but NOT x)
// - The verifier checks: "does this proof correspond to SOME x where x²=25?"
// - The answer is YES, but the verifier still doesn't know if x=5 or x=-5
// - (In field arithmetic, both 5 and the modular inverse satisfy x²=25)

Real-World ZK Applications in Web3

Privacy: Tornado Cash (shield ETH transfers), Aztec Protocol (shielded DeFi), Zcash (private transactions)

Scaling (zkRollups): zkSync Era, Scroll, Polygon zkEVM, StarkNet — process 10,000+ txs off-chain, submit one proof on-chain

Identity: Proof of humanity, proof of credentials (age > 18, KYC'd, accredited investor) without revealing PII

Cross-chain bridges: Prove state transitions on L1 without relaying all data

The Trusted Setup Problem

Some ZK systems (zk-SNARKs) require a "trusted setup" — a ceremony where secret randomness is generated and then destroyed. If the secret is NOT destroyed, whoever has it can forge proofs. Zcash's original ceremony involved multiple people independently generating randomness and destroying their devices. zk-STARKs eliminate this requirement entirely, which is why they're considered more secure long-term.

Track C: Intro to L2 Scaling (Optimism/Arbitrum Internals)

Learning Objective

Understand how L2s work under the hood, the difference between optimistic rollups and zk-rollups, and why deploying to an L2 is not just "using a different RPC URL." This knowledge is critical for every EVM engineer because 90%+ of new protocol deployments go to L2s.

// HOW OPTIMISTIC ROLLUPS WORK (Optimism, Arbitrum, Base):
//
// 1. Sequencer collects transactions off-chain, orders them, executes them
//    (users send txs to sequencer's RPC — feels instant, ~100ms confirmation)
//
// 2. Sequencer batches transactions and submits compressed batch to L1
//    (every 1-2 seconds: posts tx data + state diff to Ethereum as calldata)
//
// 3. Batch is "optimistically" accepted — assumed valid
//    7-day challenge window: anyone can submit a fraud proof if batch is invalid
//
// 4. After 7 days with no challenge: batch is FINAL on L1
//    L2 state is now as secure as Ethereum itself
//
// HOW ZK ROLLUPS WORK (zkSync, Scroll, Polygon zkEVM):
//
// 1. Operator executes transactions off-chain
//
// 2. Operator generates a ZK proof that ALL transactions in the batch are valid
//    (this takes computational effort — hence the "zk" part)
//
// 3. Submits proof + state diff to L1
//    Verifier contract checks proof in ~10ms
//    If proof is valid → batch is FINAL immediately (no challenge window)
//
// TRADEOFFS:
// Optimistic: simpler, faster proof generation, but 7-day withdrawal delay
// ZK: complex, slower proof generation, but instant finality
L2 Gas Costs Are 10-100x Lower — But Why?

On Ethereum L1, you pay for: execution gas + calldata gas + base fee. On an L2, you pay for: execution gas ONLY (in L2's native token). The calldata posting to L1 is batched across thousands of users — your share is ~$0.001. A Uniswap swap that costs $3 on L1 costs $0.03 on Base. But the Solidity code is 95% identical. Your EVM knowledge transfers directly.

L2 Security Considerations You MUST Know

• Sequencer downtime: If the sequencer goes down, you can't submit transactions (but L2 state is safe — it's all on L1). Forced transaction mechanisms exist as fallback.

• 7-day withdrawal on Optimistic L2s: Even after your tx confirms in 2 seconds, the funds aren't "fully" final for 7 days. L2 bridges mitigate this with third-party "fast exits."

• Block.timestamp differs from L1: L2 timestamps are set by the sequencer, not by L1 miners. Don't use block.timestamp for security-critical time checks on L2s.

• Different precompiles: Some L2s have different precompile addresses. Always check the target chain's precompile list.

Lesson 11.3~4 hours

The Self-Audit: Fuzzing Your Own Project

Self-Audit Protocol: The 4-Pass Method

Pass 1: Unit Test Completeness (1 hour)
For every public/external function: does it have (a) happy path test, (b) revert test for each require/custom error, (c) edge case test (zero amounts, max values, same address)? Run forge coverage. If any line is uncovered, write a test for it.

Pass 2: Fuzz Testing (1.5 hours)
For every function that takes user input: write a fuzz test with 10,000+ runs. Focus on: arithmetic functions (amount calculations), state-modifying functions (deposits, borrows), external calls (token transfers, oracle reads). Check vm.assume() rejection rate — if >50%, restructure.

Pass 3: Invariant Testing (1 hour)
Define 5+ invariants: "sum of deposits ≥ sum of withdrawals," "health factor is always accurate," "interest only accrues forward," "no user can have negative balance," "total debt never exceeds total collateral × LTV." Write handler + invariant tests. Run 10,000 sequences.

Pass 4: Adversarial Review (30 min)
Read your own code as if you're an attacker. For every external call: "Can this reenter? Can this return manipulated data? Can this be front-run?" Document every potential issue, even if you decide it's acceptable risk.

The Bias Problem: You Can't Find Your Own Bugs

Studies show that code authors find 30-50% fewer bugs than independent reviewers. You wrote the code with certain assumptions — you'll read it with those same assumptions. Mitigation: after your self-audit, ask a peer (from a Discord, Telegram, or this course's community) to review your security document. Even 15 minutes of external review catches bugs you spent hours missing.

Hands-On Task: The 4-Pass Self-Audit

Execute all 4 passes. Document: (1) coverage report screenshot, (2) fuzz test results (runs, failures found), (3) invariant test results, (4) adversarial review findings list (even "accepted risks"). Add this document to your repo's /docs/security/ folder.

Expected Outcome: /docs/security/self-audit.md with all 4 passes documented. Any bugs found during fuzzing or invariant testing should be fixed and committed before proceeding.

Rule 1: Never name your number first.

"I understand your budget is $50/hr. I can work within that for a reduced scope — for example, I'd skip the frontend integration and security report and focus on contract development only. Would that work for you?" Don't negotiate down — negotiate scope.

Rule 5: Never negotiate against yourself.
When they say "That's above our budget," say "I understand" and STOP TALKING. The silence is uncomfortable but powerful. Let them counter. People who fill silences are the ones who lose negotiations.

Chapter 14 Professional Maturity · Required Reading

The Ethical Architect

Ethics in systems engineering is not an abstract philosophy—it is a strict design constraint. In an environment where code is law, poor ethical design is a systemic vulnerability. This chapter elevates you from a developer who can compile smart contracts into an architect trusted with user funds, privacy, and critical infrastructure. Ethical rigor is your ultimate competitive advantage.

Lesson 14.1~2 hours

Clarity & Fairness

Learning Objective

Engineer systems where ambiguity is structurally eliminated through deterministic logic. Understand how digital signatures and if-then execution paths serve as modern cryptographic consent, and why all infrastructure costs must be explicitly disclosed.

Deep Explanation: Immutability as Binding Consent

Smart contracts are deterministic truth machines. Once deployed, the if-then logic executes exactly as written, removing human ambiguity. When a user signs a transaction (via ECDSA), they are providing cryptographic consent to a highly specific, mathematically bounded execution path. Ethical architecture demands that this execution path contains zero hidden branches.

In automated market makers, Service & Liquidity Provision Fees must be explicitly coded and transparently displayed. Much like a traditional currency exchange or brokerage service, these fees are direct payments for infrastructure maintenance and continuous liquidity availability—they are not opaque extraction mechanisms. If the fee calculation deviates from the frontend display, the system violates user consent.

Similarly, Real-World Data Integrity Systems (Oracles) form the foundation of fair execution. In a Collateralized Risk Management System, accurate data equals fair liquidations. If the data pipeline is opaque or manipulable, the system cannot execute fairly, breaking the fundamental trust assumption of the protocol.

Resilient System

An AMM explicitly hardcodes a 0.3% service fee in the `getAmountOut` math. The oracle integration uses aggregated, time-weighted data with strict freshness checks to guarantee execution fairness.

Fragile System

A contract dynamically adjusts fees based on an opaque off-chain signal, or relies on a single DEX's unverified spot price as an oracle, allowing silent execution manipulation.

What Goes Wrong in Production

Users sign a transaction expecting a 0.3% infrastructure fee, but a hidden `feeTo` address or dynamic fee modifier silently extracts 5%. Because the bytecode was technically verified but the math was obfuscated, users didn't realize until millions in value was drained. The system functioned "correctly" but structurally violated cryptographic consent.

How Senior Engineers Think

Senior architects assume users will *not* read the source code. They design systems where the frontend explicitly displays the exact fee, and the smart contract strictly enforces that the UI's claim perfectly matches the bytecode's execution. Any mismatch between frontend display and backend execution is treated as a critical severity exploit.

Hands-On Task

Write a `FeeDisclosure` library that calculates and returns a struct containing: `baseFee`, `protocolFee`, and `totalExecutionCost`. Write a fuzz test that mathematically proves the `totalExecutionCost` returned by the contract perfectly mirrors the deduction in `getAmountOut` under all possible input states.

Expected Outcome: A reusable Solidity library that forces any integrated system to expose its exact fee structure on-chain, cryptographically guaranteeing frontend-to-backend alignment.

Lesson 14.2~1.5 hours

Trustworthy Code

Learning Objective

Understand why hidden admin controls and opaque upgrade paths are structural vulnerabilities. Implement systems that fully disclose their capability sets and enforce responsible vulnerability disclosure.

Deep Explanation: The Attack Surface of Privileged Keys

A smart contract's trustworthiness is inversely proportional to the undisclosed power of its admin keys. If an `owner()` can pause contracts, alter fee structures, or blacklist users without these capabilities being explicitly surfaced to the user before transaction signing, the system is deceptive. Transparent contract behavior requires the ABI and NatSpec to explicitly declare all privileged operations. Furthermore, governance mechanisms (like Timelocks) must ensure users have a guaranteed window to withdraw funds before any architectural changes take effect. Vulnerability disclosure is equally critical—building exploit tooling to extort protocols rather than reporting flaws is a severe breach of professional responsibility.

Resilient System

Admin functions are gated behind a 48-hour Timelock. All capabilities are queryable via a `getCapabilities()` view function. Emergency pauses emit public events and automatically unpause after 24 hours.

Fragile System

Contract contains an undocumented `setFee(uint256)` function callable instantly by `owner()`, allowing silent extraction of user liquidity without frontend notification.

Ethical Failure Case: The Rugpull by Design

Numerous projects have deployed perfectly functional DeFi protocols that included a hidden `withdrawAll()` function in the implementation logic. When the admin executed it, the system worked exactly as coded. The ethical failure was the intentional omission of this capability from the ABI documentation, NatSpec comments, and frontend UI.

How Senior Engineers Think

Treat every `onlyOwner` modifier as a potential attack vector. During audits, senior engineers map every privileged function and cross-reference it with the project's public documentation. If a function exists in the bytecode but is not clearly disclosed in the user-facing interface, it is flagged as a high-severity deception vulnerability.

Hands-On Task

Write a `TransparencyRegistry` contract. It should automatically intercept any `onlyOwner` call, log the function signature, caller, and arguments into a public, immutable array, and require a 24-hour delay before execution. Write a view function allowing any user to audit the full history of admin interventions.

Expected Outcome: An append-only, on-chain audit log for admin actions that frontend interfaces can index, guaranteeing users can verify exactly how the protocol is being governed.

Lesson 14.3~2.5 hours

Responsibility of Custody

Learning Objective

Internalize the absolute duty of care required when engineering systems that hold user funds. Understand how collateralized risk management must be structured to prevent systemic collapse, and how zero-capital efficiency tools must be constrained.

Deep Explanation: The Mathematics of Systemic Safety

When you build Collateralized Risk Management Systems (lending protocols), you are constructing the financial guardrails for user capital. The Health Factor calculation and liquidation logic are not just mathematical formulas—they are the primary defense mechanisms against systemic insolvency. If your liquidation threshold is miscalculated by a single basis point due to an oracle decimal mismatch, the protocol becomes fundamentally broken. These mechanisms are foundational for building alternative financial systems beyond traditional interest-based models, but they demand absolute, fault-tolerant mathematical rigor.

Conversely, Zero-Capital Efficiency Tools (flash loans) are designed strictly for system health—enabling cross-pool price correction (arbitrage) and atomic portfolio rebalancing without requiring users to lock massive amounts of capital.

Critical Engineering Warning

Using zero-capital tools to exploit poorly designed protocols—draining their reserves through reentrancy or oracle manipulation—is not "clever engineering." It is a destructive act that harms users and destabilizes the ecosystem. An ethical architect builds systems resilient to these tools; they do not weaponize them to cause harm.

Resilient System

Lending protocol uses strict overcollateralization, aggregated oracles with fallbacks, and time-locked parameter changes. Flash loans are utilized by MEV keepers to correct price disparities across DEXes.

Fragile System

Protocol allows undercollateralized flash-loan-backed borrowing to manipulate its own governance voting, or uses unverified spot prices allowing self-funding liquidation attacks.

What Goes Wrong in Production

A lending protocol miscalculates the Health Factor by failing to account for decimal differences between the collateral token (18 decimals) and the debt token (6 decimals). An attacker uses a flash loan to slightly skew the collateral price, dropping the HF below 1.0, and liquidates the entire protocol's solvency. The consequence of poor engineering in custody systems is total capital destruction.

How Senior Engineers Think

When building custody systems, assume every external protocol interacting with yours is actively adversarial. Senior engineers implement strict ERC-4626 tokenized vault standards for accounting, ensuring that even if a complex flash loan interaction occurs, the share price math remains invariant and mathematically impossible to manipulate.

Hands-On Task

Build a `SafeCustody` wrapper around an existing ERC-20. Implement a strict `withdraw` function that requires a 24-hour delay via a Timelock, emits a `WithdrawalRequested` event, and allows the user to cancel within that window. Write a fuzz test proving the funds cannot be drained instantly, even by the contract owner.

Expected Outcome: A custody module where user funds are cryptographically guaranteed to be safe from instant extraction, fundamentally separating custody from admin control.

Lesson 14.4~1.5 hours

Value vs. Extraction

Learning Objective

Distinguish between systems that generate real, sustainable utility and those designed to extract value through artificial hype, mathematical obfuscation, or psychological manipulation.

Deep Explanation: Mechanism Design Intent

Engineering is fundamentally about solving problems efficiently. In Web3, you must rigorously evaluate the mechanism design intent: Does this system reduce friction, increase throughput, or provide verifiable ownership? Or does it rely on complex tokenomics that mathematically guarantee early adopters extract value from late adopters? Value creation systems have transparent, sustainable fee models (like infrastructure service fees). Extractive systems rely on artificial scarcity, opaque token emission schedules, and gamified interfaces designed to trigger impulsive user behavior.

Value Creation

A decentralized compute network where users pay micro-transactions for verifiable CPU/GPU usage. The token derives value strictly from burned utility and network demand.

Value Extraction

A protocol that requires users to buy and stake a governance token to access basic swapping functionality, creating artificial demand solely to enrich token insiders.

Ethical Failure Case: The Vampire Attack

A protocol launches with massive, unsustainable yield incentives funded entirely by inflationary token printing. Users migrate liquidity, causing competing protocols to fail. Once the competitor is dead, the attacking protocol turns off the rewards and extracts the now-captured liquidity. The engineering was sound; the intent was purely destructive extraction.

How Senior Engineers Think

Before writing a line of code, senior architects ask: "Could this exact feature be built without a token?" If the answer is yes, the token is almost certainly an extraction mechanism. If the token is required for decentralized bootstrapping, its emission curve must be mathematically bounded, transparent, and enforced on-chain.

Hands-On Task

Perform a "Token Necessity Audit" on an active protocol's whitepaper. Document: (1) What core utility does the token provide? (2) Could this utility be achieved with stablecoins or pure accounting? (3) Who controls the emission schedule, and is it immutable? Write a 1-page technical summary of your findings.

Expected Outcome: A structured analytical framework you can use to evaluate the architectural integrity and ethical foundation of any protocol you are asked to build or audit.

Lesson 14.5~1.5 hours

Privacy Rights

Learning Objective

Implement strict data minimization in smart contract design. Understand how Proof of Ownership Technology and Self-Sovereign Identity (SSI) can protect user data without relying on centralized trust.

Deep Explanation: Data Minimization on Public Ledgers

The default state of a blockchain is total, permanent transparency. Publishing raw user data (balances, identities, transaction histories) to a public mempool is a structural privacy failure. Ethical architects practice strict data minimization: storing only what is cryptographically necessary for state transitions.

Proof of Ownership Technology (often associated with NFTs) has profound real-world use cases here—acting as verifiable credentials, identity documents, and access rights without exposing the underlying personal data. By combining Zero-Knowledge proofs with Self-Sovereign Identity (DIDs), users can prove they hold a specific credential without revealing their identity. The contract verifies the cryptographic proof, not the data.

Building systems that speculate on ownership certificates is a failure of architectural intent. The engineering value lies in using these certificates for identity, records, and credentials.

Resilient System

A ZK-based voting system where the smart contract only receives a cryptographic nullifier and a proof of valid credential status, storing zero personal data on-chain.

Fragile System

A "Know Your Customer" (KYC) system that stores user passport hashes, home addresses, and full legal names directly in public contract storage.

What Goes Wrong in Production

A healthcare protocol stores encrypted medical records on-chain. While encrypted, the metadata (who uploaded, when, file size, IPFS hash) creates a highly accurate map of user medical timelines. This metadata is later scraped by data brokers, completely defeating the purpose of the encryption. If metadata exists, it will be weaponized.

How Senior Engineers Think

Senior engineers treat public blockchains as hostile environments for Personally Identifiable Information (PII). If a design requires storing user data on-chain, the design is fundamentally flawed. They push relentlessly for off-chain data storage (IPFS, encrypted local vaults) with on-chain commitments (hashes or ZK proofs).

Hands-On Task

Write a `CredentialVerifier` contract. It should accept a mocked ZK proof that proves a user holds a specific credential. The contract must emit a `CredentialVerified(address)` event without storing *any* information about the user's identity—only a boolean status mapped to a nullifier to prevent double-use.

Expected Outcome: A contract that successfully verifies user status while leaving zero traceable personal data on the public ledger, proving privacy is possible on-chain.

Lesson 14.6~1.5 hours

Data Sovereignty

Learning Objective

Engineer systems where users fundamentally own and control their identity state, mitigating the systemic risks of centralized data monopolies.

Deep Explanation: The Architecture of Self-Control

In traditional systems, identity is leased to you by a corporation (Google, Meta). If they revoke your access, your digital life ceases to exist. Decentralized Identity (DID) systems use the blockchain strictly as an immutable root of trust for public keys, but the data itself remains in the user's possession (e.g., a local mobile vault). The smart contract's role is limited to resolving the public key to a DID document and verifying signatures. It must never store the identity data itself.

User-Owned System

An ERC-725 Identity contract where the user holds the management keys. The contract acts purely as a resolver for claims, holding zero user data.

Centralized Control

A "Web3 Social" app that stores user profile pictures, bios, and social graphs directly in contract storage, making users fully dependent on the protocol's continued operation.

Ethical Failure Case: The Hostile Migration

A Web3 domain provider marketed itself as "user-owned domains." However, the registry contract included an administrative function to seize domains. When a community faction displeased the founders, their domains were instantly confiscated. Users learned too late that "on-chain" does not automatically mean "user-controlled."

How Senior Engineers Think

True data sovereignty means the protocol must be able to go to zero (all validators drop off, all relayers shut down) and the user must still be able to cryptographically prove their identity and export their data using only their private key. If a system requires the protocol's active infrastructure for the user to access their identity, it is not sovereign.

Hands-On Task

Implement a minimal DID Registry. Users register a `namehash` pointing to a `documentCID` (stored off-chain). The critical requirement: Write a modifier `onlyIdentityOwner` that allows a user to update their CID only if they provide an ECDSA signature proving control of the underlying identity key, regardless of who submits the transaction.

Expected Outcome: A registry where the protocol cannot alter user identities, and identity updates require cryptographic proof of ownership, ensuring true data sovereignty.

Lesson 14.7~1 hour

Environmental Duty

Learning Objective

Evaluate consensus mechanisms and scaling solutions as strict engineering optimization problems. Understand how Efficiency & Resource Optimization directly correlates to lower costs and reduced systemic waste.

Deep Explanation: Throughput as an Ethical Metric

Proof-of-Work (PoW) secures networks by expending physical energy. Proof-of-Stake (PoS) secures networks by expending economic capital. As an architect, choosing to build on a PoS chain, or better yet, an L2 rollup, is an ethical decision regarding resource allocation. Efficiency & Resource Optimization through L2 scaling bundles thousands of transactions into a single L1 post, drastically reducing the per-transaction energy footprint and lowering costs for end users. Engineering efficient code (tight storage packing, minimal state writes, calldata compression) directly reduces the computational waste of the global network.

Optimized Architecture

Deploying a high-frequency application on an Optimistic or ZK-Rollup (L2), using batched operations to minimize L1 data posting and maximize gas efficiency.

Wasteful Architecture

Deploying a simple accounting app on a PoW chain, using unoptimized loops and dynamic arrays that consume 5x the necessary gas per transaction.

What Goes Wrong in Production

A popular NFT minting contract was written so inefficiently (using loops for IPFS hash storage instead of mappings) that the gas required per mint was equivalent to an average household's daily energy consumption. The resulting network congestion and backlash damaged the project irreparably. Poor code isn't just expensive; it is systemically wasteful.

How Senior Engineers Think

Gas optimization is not just about saving the user money; it is an act of reducing the network's aggregate computational load. Senior engineers profile their contract's bytecode size and execution traces, treating high gas consumption as a system inefficiency that harms the entire ecosystem's sustainability.

Hands-On Task

Take a poorly written contract (e.g., one that uses a loop to iterate through an array of users to distribute rewards). Refactor it to use the O(1) Synthetix reward pattern. Run `forge test --gas-report` on both versions. Calculate the percentage reduction in gas units and extrapolate the network resource savings over 100,000 transactions.

Expected Outcome: A concrete, quantified technical report demonstrating how architectural choices directly scale down network resource consumption.

Social Impact

Learning Objective

Deep Explanation: Transparency as Access

Billions of people lack access to traditional banking infrastructure. Web3 allows anyone with a cheap smartphone to participate in global finance. However, financial inclusion only works if the systems are cheap (hence L2 scaling) and transparent. In charity and aid, traditional systems lose up to 40% of funds to intermediary overhead and corruption. A transparent charity system uses smart contracts to lock donated funds, automatically release them upon verifiable on-chain milestones (via Real-World Data Integrity Systems), and publicly track every transaction. This structurally reduces dependency on corruptible intermediaries.

Impactful System

An aid distribution contract that releases funds only when a verified supply chain oracle confirms food delivery to a specific geographic coordinate.

Exploitative System

A "micro-finance" app targeting developing nations with 300% APR collateralized loans, facilitated by hidden fees embedded in the oracle exchange rate.

Marketing "financial inclusion" to unbanked populations while hiding extreme fee structures behind complex UIs. Users in developing nations lose their meager collateral because they did not understand the liquidation threshold or the oracle spread. Inclusion without absolute clarity is exploitation.

The system should default to the safest parameters (high overcollateralization, slow withdrawals) rather than the most capital-efficient ones. Safety is an architectural requirement, not a user preference.
